─────────────────────────────────────────────
1) UserDto
   파일: dto/UserDto.java
─────────────────────────────────────────────
  필드:
    - userId        : String  — 로그인 아이디 (4-20자)
    - password      : String  — 원본 비밀번호 (암호화 전)
    - nickname      : String  — 닉네임 (회원가입 전용)
    - email         : String  — 이메일 (회원가입 전용)
    - phoneNumber   : String  — 전화번호 (회원가입 전용)

  역할:
    회원가입(POST /api/user/signup)과 로그인(POST /api/user/login) 요청 시
    프론트엔드에서 전송하는 JSON 데이터를 담는 그릇 역할.
    로그인 시에는 userId, password만 사용하고 나머지 필드는 무시됨.

  ★ 없으면 발생하는 문제:
    - @RequestBody 로 요청 JSON을 파싱할 수 없어 컴파일/런타임 오류 발생.
    - Controller에서 개별 @RequestParam으로 받아야 하므로 코드 복잡도 급증.
    - 요청 구조가 변경될 때 Controller 코드 전체를 수정해야 함.
    - 비밀번호 같은 민감 정보가 Entity 객체를 통해 직접 노출될 위험이 있음.

─────────────────────────────────────────────
2) BackofficeUserDto
   파일: dto/BackofficeUserDto.java
─────────────────────────────────────────────
  필드:
    - id       : String  — 관리자 로그인 아이디
    - password : String  — 원본 비밀번호
    - level    : Integer — 권한 레벨 (1=일반, 2=슈퍼; 로그인 시 null)

  역할:
    백오피스 관리자 가입(POST /api/backoffice/signup)과
    로그인(POST /api/backoffice/login) 요청 처리용 DTO.
    UserDto와 분리하여 관리자 전용 필드(level)를 독립적으로 관리.

  ★ 없으면 발생하는 문제:
    - UserDto와 공유 시 관리자 전용 필드(level)가 일반 회원 DTO에 혼입.
    - 관리자/일반 회원 인증 로직 구분이 불명확해져 보안 취약점 발생 가능.
    - 두 도메인의 요청 스펙이 달라질 때 한쪽 수정이 다른 쪽에 영향을 줌.

─────────────────────────────────────────────
3) ErrorLogDto
   파일: dto/ErrorLogDto.java
─────────────────────────────────────────────
  필드:
    - errorSource   : String  — 에러 발생 위치 (FRONTEND / BACKEND)
    - errorType     : String  — 에러 유형 (NETWORK / VALIDATION / SERVER 등)
    - errorMessage  : String  — 에러 메시지
    - errorDetails  : String  — 상세 정보 (스택 트레이스)
    - statusCode    : Integer — HTTP 상태 코드
    - requestUrl    : String  — 에러 발생 URL
    - httpMethod    : String  — HTTP 메서드
    - requestBody   : String  — 요청 본문
    - userId        : String  — 에러 발생 시 로그인한 사용자 ID
    - userAgent     : String  — 사용자 브라우저 정보
    - ipAddress     : String  — 사용자 IP (백엔드에서 채움)

  역할:
    프론트엔드에서 에러 발생 시 POST /api/error-log 로 전송되는 에러 정보를
    구조화하여 DB(error_log 테이블)에 저장하기 위한 DTO.

  ★ 없으면 발생하는 문제:
    - 프론트엔드 에러 수신 엔드포인트가 파싱 불가 → 에러 로그 수집 불능.
    - 에러 정보가 단순 문자열로만 저장되어 분석/검색이 불가능해짐.
    - 디버깅, 운영 모니터링, 장애 추적에 필요한 정형화된 로그 부재.

4) RestaurantDto
   파일: dto/RestaurantDto.java
─────────────────────────────────────────────
  필드:
    - name        : String       — 점포명
    - category    : String       — 카테고리
    - address     : String       — 상세 주소
    - location    : String       — 지역명
    - phone       : String       — 전화번호
    - priceRange  : String       — 가격대 (₩~₩₩₩₩)
    - description : String       — 소개글
    - imageUrl    : String       — 이미지 URL (현재 null)
    - hashtags    : List<String> — 등록할 해시태그 이름 목록

  역할:
    점포 등록(POST /api/backoffice/restaurant/register)과
    점포 수정(PATCH /api/backoffice/restaurant/{idx}) 요청 DTO.
    Entity와 분리하여 DB 자동 생성 컬럼(idx, regDate 등)이 외부 노출되지 않도록 보호.

  ★ 없으면 발생하는 문제:
    - restaurant Entity를 직접 @RequestBody로 받게 되면 idx, state, regDate 등을
      외부에서 임의 주입 가능 → 데이터 무결성 파괴 위험 (Mass Assignment 취약점).
    - 등록/수정에 공통 필드 외 추가 필드(hashtags) 관리가 불가능해짐.

     경로: repository/
     * Spring Data JPA — JpaRepository를 상속한 인터페이스
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

─────────────────────────────────────────────
1) UserInfoRepository
   파일: repository/UserInfoRepository.java
─────────────────────────────────────────────
  주요 메서드:
    - findByUserId(String)          : 아이디로 회원 조회
    - existsByUserId(String)        : 아이디 중복 확인
    - findByEmail(String)           : 이메일로 조회
    - existsByEmail(String)         : 이메일 중복 확인
    - findByPhoneNumber(String)     : 전화번호로 조회
    - existsByPhoneNumber(String)   : 전화번호 중복 확인
    - findAllByOrderByRegDateDesc() : 최신 가입순 전체 조회
    - findByUserIdContainingIgnoreCase...() : 아이디·닉네임 검색

  역할:
    user_info 테이블에 대한 모든 DB 접근을 단일 창구로 집중.
    JPA의 쿼리 메서드 네이밍 규칙으로 SQL 직접 작성 없이 쿼리 생성.

  ★ 없으면 발생하는 문제:
    - Service에서 EntityManager를 직접 사용해야 하므로 코드 양이 10배 이상 증가.
    - 중복 체크 로직을 Service에서 직접 JPQL로 구현해야 함.
    - 테스트 시 Repository를 Mock으로 대체할 수 없어 통합 테스트만 가능.

2) RestaurantRepository
   파일: repository/RestaurantRepository.java

  주요 메서드:

    - 동일 이름+주소 점포 중복 등록 방지 로직을 Service가 직접 쿼리 작성해야 함.
    - 활성/비활성 점포 분리 조회 불가 → 비활성 점포가 메인 페이지에 노출될 수 있음.

4) RestaurantHashtagRepository
   파일: repository/RestaurantHashtagRepository.java
─────────────────────────────────────────────
  주요 메서드:
    - findByRestaurantEntityIdxAndState(...)             : 점포별 활성 태그
    - findActiveHashtagsWithTagByRestaurantIdx(...)      : JOIN FETCH 단일 점포 태그
    - findActiveHashtagsByRestaurantIdxIn(...)           : IN절 일괄 태그 조회
    - existsByRestaurantEntityIdxAndHashtagIdx(...)      : 태그 중복 연결 확인

  역할:
    restaurant_hashtag 중간 테이블 접근.
    JOIN FETCH로 Lazy 로딩을 방지하고, IN절 일괄 조회로 메인 페이지 N+1 문제 해결.

  ★ 없으면 발생하는 문제:
    - 점포 수만큼 해시태그 쿼리가 추가 발생 (N+1 문제) → 성능 저하.
    - 동일 점포에 같은 태그 중복 등록 방지 불가.
    - 메인 페이지 목록 로딩 시 DB 쿼리 폭발적 증가.

    - 별점 순 정렬 기능 동작 불가.
    - N개 점포에 대해 N번 쿼리 실행 필요 (성능 문제).

  ★ 없으면 발생하는 문제:
    - 수정 성공 후 표시할 점포명(name)을 알 수 없음.

─────────────────────────────────────────────
  필드: restaurantHashtagIdx, hashtagIdx, name, state, regDate

        인라인 수정/삭제 시 어느 인덱스를 사용할지 명확히 구분.

  ★ 없으면 발생하는 문제:
    - 삭제 API(restaurant_hashtag.idx)와 수정 API(hashtag.idx)를 혼동 가능.

2) HashtagListApiResponse / HashtagMutationApiResponse
─────────────────────────────────────────────

  ★ 없으면 발생하는 문제:
    - 등록/삭제 성공 여부 불명확 → 로컬 상태 업데이트 조건 분기 불가.

─────────────────────────────────────────────
3) HashtagMasterItem / HashtagMasterListApiResponse
─────────────────────────────────────────────
  필드: idx, name, useCount, regDate / success, data[], total

  역할: 백오피스 해시태그 마스터 목록 페이지 데이터 타입.
        useCount로 사용 비율 시각화(프로그레스 바) 계산.

  ★ 없으면 발생하는 문제:
    - useCount가 string인지 number인지 불명확 → 최대값 계산 시 오류.

  [프론트 요청 DTO → 백엔드 수신 DTO]
  RestaurantRegisterFormData  →  RestaurantDto          (점포 등록)

  HashtagItem.restaurantHashtagIdx → DELETE /restaurant/hashtag/{idx} (태그 삭제)

  [백엔드 응답 DTO → 프론트 수신 타입]
  RestaurantListItemDto       →  RestaurantListItem      (백오피스 목록)
  PublicRestaurantDto         →  PublicRestaurantItem    (공개 메인 페이지)
  HashtagResponseDto          →  HashtagItem             (해시태그 모달)
  HashtagMasterDto            →  HashtagMasterItem       (마스터 목록)
  UserListItemDto             →  UserListItem            (회원 목록)

  [DB Entity → 백엔드 DTO 변환 지점]
  restaurant      →  RestaurantListItemDto / PublicRestaurantDto
  UserInfo        →  UserListItemDto  (password 필드 제외 핵심)
  RestaurantHashtag + Hashtag  →  HashtagResponseDto / HashtagMasterDto
